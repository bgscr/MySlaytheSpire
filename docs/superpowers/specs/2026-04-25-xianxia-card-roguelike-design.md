# 东方玄幻卡牌肉鸽设计规格

日期：2026-04-25

## 目标

创建一款运行在 Windows 客户端上的 2D 卡牌构筑肉鸽游戏。创意上只参考成熟卡牌肉鸽类型框架，不复刻《杀戮尖塔》或其他具体作品的角色、机制命名、美术、UI、事件文本或内容表达。项目题材为东方玄幻/修行，首版目标是一个内容较少但完整的一章级系统原型，并为后续玩法、内容、工具、测试、平台和发布流程扩展留出清晰边界。

首个版本必须验证这些问题：

- 跑图、战斗、奖励、构筑、事件、商店、Boss、失败/通关结算是否形成完整闭环。
- 剑修和丹修是否有清晰差异。
- Godot Resource 与组合式脚本是否足以支撑后续批量扩内容。
- 高表现力战斗反馈是否能与核心逻辑解耦。
- 存档、seed、平台抽象、本地化、调试工具是否从一开始支持长期扩展。

## 目标平台和技术栈

- 引擎：Godot 4.6.2-stable。
- 目标平台：Windows 客户端，优先 Win11；Win10 在成本低时兼容。
- 代码语言：GDScript 为主；复杂算法、批量工具或编辑器插件未来再评估 C#。
- 运行模式：首版完全单机，本地运行和本地 Windows 导出包优先。
- CI：首版暂缓，项目稳定后再接完整发布流水线。
- 文本：首版中文优先，但从一开始使用 Godot gettext `.po` 本地化流程。

Godot 版本锁定依据：2026-04-25 核对官方信息时，Godot 4.6.2-stable 是当前稳定维护版本，Godot 4.7 仍处于开发预览线。

## 产品范围

首版为完整一章级系统原型，不是灰盒技术 demo，也不是正式商业内容量。

首版内容目标：

- 2 个首发角色：剑修、丹修。
- 每个角色约 20 张专属牌。
- 15-20 个遗物。
- 10+ 普通/精英敌人。
- 2 个 Boss。
- 完整事件池。
- 商店、奖励、地图路线、失败结算、通关结算。
- 完整局内存档和继续游戏。
- 可复现 seed。
- 高表现力战斗反馈。
- 面向未来 Steam/商店发行的接口预留，但首版不接 Steamworks SDK。

核心循环：

```text
主菜单
-> 新局/继续游戏
-> 选角色/设置 seed
-> 路线地图
-> 节点遭遇：普通战 / 精英 / 事件 / 商店 / Boss
-> 奖励和构筑变化
-> 下一节点
-> 失败结算或通关结算
```

## 顶层扩展原则

项目从首版开始按长期扩展设计。扩展性不只指内容数量，也包括玩法规则、资源生产、调试工具、测试、导出、平台服务和发布流程。

设计约束：

- 玩法状态和表现层解耦。
- 内容定义和运行实例解耦。
- 平台能力和游戏逻辑解耦。
- 新内容优先通过 Resource 和组合组件新增，不改核心流程。
- 新角色复用 Run、Combat、Reward、Map、Save 基础设施。
- 新节点类型通过节点处理器扩展，不改地图生成核心。
- 新平台能力通过 PlatformService 接口扩展，不污染玩法代码。
- 自动化构建、release draft、changelog、版本号、导出包等流程后续可接入，不要求首版实现。

## 架构模块

### Run 模块

Run 模块管理一局游戏的顶层状态：

- 当前 seed。
- 当前角色。
- 当前血量、最大血量、金币。
- 当前牌组。
- 当前遗物。
- 当前地图。
- 当前节点和已访问节点。
- 当前奖励、事件、商店或战斗状态。
- 失败、通关、暂停、继续游戏状态。

Run 模块是存档的主要入口。其他模块不直接写磁盘，而是把运行状态交给 Save 模块序列化。

### Combat 模块

Combat 模块管理回合制战斗逻辑：

- 抽牌堆、手牌、弃牌堆、消耗区。
- 能量、回合数、行动队列。
- 玩家状态和敌人状态。
- 敌人意图和敌人行动。
- 伤害、护体/格挡、治疗、状态施加、状态衰减。
- 卡牌打出、目标选择、效果解析和结算。

Combat 模块只产生 CombatEvent，不直接播放动画、音效或粒子。表现层订阅 CombatEvent 后播放反馈。

### Card 和 Effect 模块

卡牌定义使用 Godot Resource。卡牌 Resource 包含：

- 卡牌 id。
- 本地化文本 key。
- 费用。
- 类型。
- 稀有度。
- 目标规则。
- 标签和关键词。
- 卡图资源路径。
- 效果列表。
- 升级后的变化。

效果通过组合式 Effect、Condition、Action 资源或 GDScript 类表达。新卡牌应主要通过新增 Resource 和组合效果完成，避免每张牌写独立硬编码逻辑。

### Map 模块

Map 模块基于 seed 生成路线图。节点类型包含：

- 普通战。
- 精英。
- 事件。
- 商店。
- Boss。

后续可扩展特殊节点，例如修炼、秘宝、奇遇、心魔、锻造、休整等。节点运行逻辑通过 MapNodeHandler 扩展。

### Reward 模块

Reward 模块统一处理跑图奖励：

- 选牌。
- 遗物。
- 金币。
- 治疗。
- 升级。
- 删牌。
- 后续新增奖励类型。

Reward 模块应能根据 seed、角色、当前牌组、稀有度规则和奖励上下文生成候选项，并能被调试工具复现。

### Event 和 Shop 模块

事件使用数据定义和运行时选项解析：

- 事件 id。
- 本地化文本 key。
- 出现条件。
- 选项列表。
- 选项条件。
- 选项效果。
- 事件状态。

商店使用统一购买项模型：

- 买牌。
- 买遗物。
- 删牌。
- 治疗或升级。
- 后续特殊服务。

### Relic 模块

遗物以被动触发为主，通过事件总线监听战斗和跑图事件。遗物触发点包括但不限于：

- 战斗开始。
- 回合开始。
- 回合结束。
- 抽牌。
- 打出卡牌。
- 造成伤害。
- 受到伤害。
- 获得状态。
- 战斗胜利。
- 进入节点。
- 获得奖励。

新遗物应主要新增 Resource 和触发效果，不改 Combat 或 Run 核心。

### Save 模块

Save 模块负责完整局内存档和继续游戏。存档必须包含：

- 存档格式版本号。
- 项目版本号。
- seed。
- RunState。
- 当前牌组和卡牌实例状态。
- 抽牌堆、弃牌堆、消耗区等必要战斗状态。
- 玩家血量、金币、遗物。
- 地图、节点进度、事件状态。
- 当前战斗、奖励、商店或事件的恢复信息。

存档格式必须支持后续迁移。字段新增时应能保持旧存档至少给出明确错误或迁移路径。

### Presentation 模块

Presentation 模块负责 UI、动画、粒子、镜头、音频和转场。它订阅游戏事件，不反向驱动核心结算。

原则：

- 核心逻辑不等待动画才能正确结算。
- 动画可跳过、可调试、可降级。
- 高表现力通过事件队列和表现配置实现。
- 调试模式可以关闭慢帧、镜头震动和高强度粒子。

### Platform 模块

Platform 模块预留发行平台能力：

- 成就。
- 统计。
- 云存档。
- 平台语言。
- 每日挑战。
- 排行榜。
- 发行版本信息。

首版只实现本地 PlatformService，不接 Steamworks SDK。后续 SteamService 通过同一接口接入。

### DevTools 模块

首版实现基础调试面板：

- 修改血量。
- 修改金币。
- 抽牌。
- 添加遗物。
- 跳转节点。
- 固定 seed。
- 直接进入战斗。

后续扩展为完整内容调试工具：

- 卡牌浏览器。
- 敌人测试场。
- 事件测试器。
- 奖励池检查。
- 存档查看。
- 资源一致性检查。

## 游戏系统设计

### 战斗规则

战斗采用回合制：

- 玩家每回合抽牌并获得能量。
- 玩家通过出牌造成伤害、获得护体/格挡、施加状态、抽牌、获得资源或触发连携。
- 敌人每回合显示意图。
- 玩家结束回合后，敌人按意图行动。
- 回合结束处理状态衰减、持续伤害、延迟效果和触发器。
- 胜利进入奖励。
- 失败进入结算。

### 首发角色

剑修：

- 核心表达：剑气、连斩、破势、爆发。
- 玩法倾向：主动进攻、多牌连携、短时间爆发。
- 示例方向：打出多张攻击牌后增强、对低护体敌人追加伤害、积累剑气后释放终结。

丹修：

- 核心表达：炼丹、药性、净化、毒/灼、回复、状态转化。
- 玩法倾向：准备、转化、持续收益。
- 示例方向：把负面变资源、用药性强化卡牌、通过毒/灼和回复形成长期优势。

后备扩展职业：

- 符修：符箓布局、延迟触发、阵法组合。
- 体修：护体蓄势、反击、承伤爆发。
- 御兽：召唤灵兽、协同攻击、伙伴成长。

### 跑图系统

首版地图是一章级路线图。路线需要提供明确风险收益：

- 普通战提供基础奖励。
- 精英提供更高风险和更好遗物奖励。
- 事件提供不确定收益。
- 商店提供金币消耗和构筑修正。
- Boss 作为章节终点。

地图生成必须基于 seed，尽量保持同 seed 可复现。

### 随机性和 seed

每局有 seed。同 seed 应尽量复现：

- 地图结构。
- 节点类型分布。
- 敌人组合。
- 奖励候选。
- 事件候选。
- 商店候选。
- 关键随机结果。

随机系统应集中管理，避免模块直接使用不可控随机源。

## 表现与体验设计

### 视觉方向

美术方向为 2D 手绘/插画风，题材为东方玄幻/修行。可使用 AI/生成式资源辅助制作：

- 角色立绘。
- 敌人图。
- Boss 图。
- 卡牌插画。
- 背景。
- 图标。
- 部分特效素材。

资源必须按可替换资产管理，未来可以替换为人工精修版本。需要保留资源来源、用途和可替换路径，方便统一风格和版权审查。

UI 不直接模仿参考作品，应采用原创修行/秘境风格。可使用灵纹、玉简、丹炉、剑光、符箓、山海秘境等视觉语言。

### 战斗表现

首版高表现力目标：

- 卡牌拖拽、悬停、目标选择、打出、回手、弃牌都有动画。
- 攻击有轨迹、剑气、药雾、符光等反馈。
- 受击有闪白、震动、数字飘字、状态图标变化。
- Boss 和精英有明显出场、意图、受击、特殊行动反馈。
- 镜头震动、慢帧、粒子、屏幕特效有统一强度控制。
- 表现不能干扰读牌和理解敌人意图。

### 音频

首版先实现音频管理器和触发接口：

- BGM 通道。
- SFX 通道。
- UI 通道。
- 音量设置。
- 战斗事件音频钩子。
- UI 事件音频钩子。

具体素材可以生成或占位。后续可扩展动态音乐、Boss 音乐、事件音乐和更细粒度技能音效。

### 输入和窗口

输入策略：

- 鼠标为主。
- 键盘完整支持。
- 菜单导航、选牌、结束回合、目标切换、确认、取消、地图节点选择都要有键盘路径。

窗口策略：

- 窗口模式优先。
- UI 支持可缩放窗口和安全边距。
- 后续再细化 16:9、16:10、21:9、Steam Deck 和手柄支持。

## 本地化

首版内容中文优先，但所有 UI、卡牌、敌人、事件、遗物、奖励和设置文本都应通过本地化 key 读取。

本地化使用 Godot gettext `.po` 标准流程。未来新增语言时，通过维护对应 `.po` 文件扩展，不需要改玩法逻辑。

## 发布和平台策略

项目未来面向 Steam/商店发行，但首版不接 Steamworks SDK。

首版需要预留：

- 发行版本号。
- 构建元数据。
- 本地成就接口。
- 本地统计接口。
- 云存档接口。
- 平台语言接口。
- 每日挑战接口。
- 排行榜接口。

后续可添加：

- Steamworks 实现。
- Demo/正式版分支。
- 崩溃日志。
- GitHub Actions。
- Godot headless test。
- Windows artifact。
- release draft。
- changelog 和版本发布流程。

## 测试策略

首版测试策略为逻辑测试加场景冒烟测试。

逻辑单元测试覆盖：

- 卡牌效果。
- 状态结算。
- 伤害和护体/格挡计算。
- 遗物触发。
- 随机 seed。
- 奖励生成。
- 存档序列化和反序列化。

场景冒烟测试覆盖：

- 主菜单加载。
- 新局流程。
- 战斗场景加载和基础流转。
- 地图场景加载和节点选择。
- 奖励场景加载和选择。
- 商店场景加载。
- 事件场景加载。
- 失败/通关结算场景加载。

首版 CI 暂缓，测试先在本地运行。目录和命令应为未来 GitHub Actions 保留清晰入口。

## 内容生产策略

首版内容目标较高，因此必须分阶段制作：

1. 先完成核心闭环和少量示例内容。
2. 再扩充到两个角色的基础卡池。
3. 再补遗物、敌人、Boss、事件和商店。
4. 再补高表现力动画、特效、音频和转场。
5. 最后做存档、调试、导出和稳定性检查。

AI/生成式资源用于首版视觉完整度，但资源路径和命名必须支持后续替换。

## 风险和应对

内容范围风险：

- 风险：2 角色、40 张牌、15-20 遗物、10+ 敌人、2 Boss 和完整事件池对首版来说不小。
- 应对：先做纵向最小闭环，再扩目标内容量。每类内容先有示例，再批量补齐。

表现力风险：

- 风险：动画、粒子、镜头、音效可能拖慢核心系统。
- 应对：表现层订阅事件，不驱动逻辑。所有表现可跳过、可降级、可调试。

数据复杂度风险：

- 风险：Resource 继承层级过深会难维护。
- 应对：优先组合式 Effect、Condition、Action，减少深继承。

平台扩展风险：

- 风险：过早接 Steamworks 或联网会拖慢核心验证。
- 应对：首版只做接口和本地实现，稳定后再接真实 SDK 和服务。

版权和素材风险：

- 风险：参考作品边界不清或 AI 素材风格/版权不可控。
- 应对：只参考类型框架，所有内容原创。记录 AI 资源来源和用途，后续可替换和审查。

## 非目标

首版不做：

- 真实 Steamworks SDK 集成。
- 真实联网排行榜。
- 真实每日挑战服务。
- 完整 CI/CD 发布流水线。
- 手柄和 Steam Deck 完整适配。
- 多章节正式内容量。
- 英文或其他语言完整翻译内容。
- 商业最终美术品质。

## 验收标准

设计规格阶段完成后，进入实施计划前应满足：

- 范围、架构、核心系统、表现、工程质量、扩展性要求已确认。
- 规格文档已提交到仓库。
- 用户已审阅规格并确认可以进入实施计划。

首个可玩版本完成后应满足：

- 可在 Windows 本地运行。
- 可从主菜单开始新局并选角色。
- 可完成至少一局从地图到结算的完整流程。
- 支持继续游戏。
- 同 seed 尽量复现地图、奖励和关键随机。
- 剑修和丹修均有可玩卡池。
- 战斗、事件、商店、奖励、Boss、结算均可进入。
- 高表现力反馈具备基本完整体验。
- 基础调试面板可用。
- 核心逻辑测试和场景冒烟测试可本地运行。
