# High Presentation Foundation Design

## Goal

Build the first high-presentation combat pass for the current Godot prototype. The pass should make combat feel readable and responsive through mouse drag play, target highlighting, floating numbers, hit flashes, and status pulses while preserving the existing deterministic combat logic.

This is a foundation pass, not the final art pass. The first implementation uses a neutral development skin and programmatic placeholder effects. The architecture must still leave a clean path for later sword-qi trails, medicine mist, camera impulses, slow motion, particles, audio cues, and asset-pack-driven presentation profiles.

## Current Context

The project already has a complete one-chapter playable loop:

- Main menu, map, combat, rewards, events, shop, save/continue, run summary.
- Two characters with 20 cards each.
- Default content with 16 enemies, 20 relics, and 12 events.
- Combat rules are driven by `CombatSession`, `CombatEngine`, `EffectExecutor`, and `CombatStatusRuntime`.
- `CombatScreen` currently creates a functional but gray-box UI with buttons and labels.
- `PresentationEventRouter` exists as an early hook but only prints basic event messages.

The next risk is no longer content volume. The thin part is feel: cards do not move, targets do not visually respond, state changes are easy to miss, and enemy turns feel abrupt.

## Confirmed Design Decisions

- First priority: core combat feel package.
- Future priority: status feedback package and richer card-flow package.
- Architecture: combat presentation event queue first.
- Event envelope: lightweight GDScript class, not raw `Dictionary`.
- Visual style: neutral development skin, restrained colors and motion.
- Input: mouse drag play plus existing click fallback.
- Drag behavior: target-aware drag release.
- Assets: programmatic placeholder effects now, asset-pack integration later.
- Configuration entry: `DebugOverlay` development toggles first, no formal settings menu yet.
- Event sourcing: UI interaction events plus pre/post combatant snapshot diff.
- Coverage: player and enemy turns both produce presentation feedback.
- Playback: non-blocking, asynchronous feedback; gameplay state remains authoritative.

## Non-Goals

This pass does not include:

- Final UI skin, final card art, final enemy art, or a broad generated asset pack.
- Real audio assets.
- Real slow motion or camera shake playback, beyond future event hooks.
- Touch or controller drag input.
- Animation-driven gameplay sequencing.
- A formal player settings screen or persisted presentation settings.
- A full cinematic combat system.
- Refactoring core combat rules to know about presentation.

## Architecture

The presentation foundation adds a small event pipeline beside the existing combat flow:

```text
CombatScreen input and state observation
  -> CombatPresentationEvent
  -> CombatPresentationQueue
  -> CombatPresentationLayer
  -> Programmatic feedback nodes
```

`CombatScreen` remains the bridge between UI input and combat state. It produces interaction presentation events directly and observes combat state changes by comparing snapshots before and after gameplay actions.

`CombatSession`, `CombatEngine`, `EffectExecutor`, and `CombatStatusRuntime` remain gameplay systems. They do not import presentation classes and do not wait for animation completion.

`CombatPresentationQueue` stores, filters, drains, and clears presentation events. It is intentionally simple in the first pass so it can be unit-tested without a scene tree.

`CombatPresentationLayer` consumes drained events and plays visible feedback in the combat scene. First-pass effects are programmatic Godot UI nodes and tweens.

## Event Envelope

Create `scripts/presentation/combat_presentation_event.gd`:

```gdscript
class_name CombatPresentationEvent
extends RefCounted

var event_type: String = ""
var source_id: String = ""
var target_id: String = ""
var card_id: String = ""
var amount: int = 0
var status_id: String = ""
var text: String = ""
var intensity: float = 1.0
var tags: Array[String] = []
var payload: Dictionary = {}
```

This class is a runtime event envelope. It is deliberately not a Resource yet because first-pass events are generated by combat runtime state and UI interaction, not authored content.

Future expansion remains possible through:

- `tags` for grouping and filtering, such as `cinematic`, `camera`, `audio`, or `reduced_motion_sensitive`.
- `payload` for specialized parameters, such as trail points, particle colors, cue keys, shake vectors, or durations.
- Later adapters that interpret the same event differently for programmatic, cinematic, audio, or asset-pack profiles.
- Later Resource definitions if authored presentation assets become necessary.

## First-Pass Event Types

Implement these event types first:

```text
card_hovered
card_unhovered
card_drag_started
card_drag_moved
card_drag_released
target_highlighted
target_unhighlighted
card_played
damage_number
block_number
status_number
combatant_flash
status_badge_pulse
```

Reserve these future event types in documentation and tests where helpful, but do not implement full playback yet:

```text
cinematic_slash
particle_burst
camera_impulse
slow_motion
audio_cue
```

## Event Queue

Create `scripts/presentation/combat_presentation_queue.gd`.

The queue should support:

- `enqueue(event: CombatPresentationEvent) -> void`
- `drain() -> Array[CombatPresentationEvent]`
- `clear() -> void`
- `size() -> int`
- Optional filtering through a presentation config object or dictionary.

The first queue does not need timeline scheduling, awaitable playback, or priority sorting. It only needs deterministic FIFO behavior and clear filtering rules.

Filtering should be conservative:

- If all presentation is disabled, drop events during `enqueue`.
- If floating text is disabled, drop `damage_number`, `block_number`, and `status_number`.
- If flash is disabled, drop `combatant_flash`.
- If target highlighting is disabled, drop `target_highlighted` and `target_unhighlighted`.
- If dragging is disabled, ignore drag presentation events while keeping click fallback usable.
- If future cinematic is disabled, drop events tagged `cinematic`.

## Presentation Config

Create `scripts/presentation/combat_presentation_config.gd`:

```gdscript
class_name CombatPresentationConfig
extends RefCounted

var enabled := true
var drag_enabled := true
var floating_text_enabled := true
var flash_enabled := true
var target_highlight_enabled := true
var status_pulse_enabled := true
var cinematic_enabled := false
```

The fields are:

```text
enabled
drag_enabled
floating_text_enabled
flash_enabled
target_highlight_enabled
status_pulse_enabled
cinematic_enabled
```

The first implementation should expose these toggles in `DebugOverlay`. They are development controls, not player settings, and do not need to persist in save files.

## Presentation Layer

Create `scripts/presentation/combat_presentation_layer.gd` and instantiate it from `CombatScreen`.

Responsibilities:

- Drain the queue during `_process` or a controlled refresh point.
- Play floating labels for damage, block, and status changes.
- Flash combatant UI containers on hit or status application.
- Pulse status badge labels when status layers change.
- Apply hover and selected styling to card controls.
- Apply target highlight styling to enemy buttons and the player status region.
- Remove temporary nodes after their tween finishes.

The layer should not:

- Mutate `RunState`.
- Mutate `CombatSession` rules.
- Decide whether a card can be played.
- Block input while effects play.
- Own gameplay target selection.

## State Delta Capture

Combat result events come from snapshot comparison, not from `CombatEngine`.

Before a gameplay action:

- Capture player and enemy combatant ids.
- Capture current HP.
- Capture current block.
- Capture positive status layers.

After the gameplay action:

- HP decrease produces `damage_number` and `combatant_flash`.
- Block increase produces `block_number`.
- Status layer increase or decrease produces `status_number` and `status_badge_pulse`.
- Multiple combatants can produce events from one action.

This applies to:

- Player card play.
- Enemy turns.
- Poison ticks at turn start.
- Enemy status intents.
- Relic-triggered combat effects when they mutate combatant state.

Snapshot comparison should not guess card intent. It observes the actual state delta after rules resolve, keeping presentation separate from gameplay.

## Drag Play

Mouse drag is added as the main feel improvement. Existing click card selection and target click flow remain as fallback.

Rules:

- Enemy-target cards can be dragged onto an enemy control and released to play on that enemy.
- Player-target cards can be dragged onto the player status region and released to confirm on the player.
- Self or no-target cards can be dragged upward past a release threshold to play.
- Cards released without a valid target return to their hand position and leave click flow unchanged.
- If dragging is disabled in DebugOverlay, click flow still works exactly as before.

Drag presentation:

- `card_drag_started` creates a visual ghost or lifts the card control.
- `card_drag_moved` updates ghost position and target hover.
- `target_highlighted` marks the current valid target.
- `card_drag_released` clears the ghost and target highlight.

## Click Fallback

The existing click selection path remains required:

- Clicking an enemy-target card can still enter enemy target selection.
- Clicking an enemy can still confirm the selected card.
- Clicking a player-target card can still enter player confirmation.
- Clicking confirm/cancel controls still works.

Smoke tests should continue to exercise click fallback so drag changes do not destabilize the existing loop.

## Enemy Turn Feedback

Enemy turns should use the same queue and layer as player actions. This avoids a visible quality gap where player actions feel alive but enemy actions remain gray-box.

Expected first-pass feedback:

- Enemy attacks produce player damage numbers and flash.
- Enemy block intents produce block numbers on the enemy.
- Enemy status intents produce status numbers and status badge pulse.
- Poison ticks produce damage numbers and flash before the actor's intent.

The enemy turn should not wait for animations before continuing. Effects may overlap if the turn advances quickly; that is acceptable for this pass.

## Neutral Development Skin

Use a restrained, readable development style:

- Dark neutral combat surface.
- Subtle white, gray, teal, and warm gold accents.
- No heavy single-hue palette.
- Short, readable floating labels.
- Thin target outlines.
- Brief flash and pulse durations.
- Motion that improves readability, not spectacle.

The style should be easy to replace later with a cultivated xianxia visual language.

## Future Cinematic Path

This pass should explicitly leave hooks for later B-style cinematic effects:

- `cinematic_slash` can play sword-qi trails for sword attacks.
- `particle_burst` can play medicine mist, poison clouds, or talisman sparks.
- `camera_impulse` can add shake or hit-stop intensity.
- `slow_motion` can briefly adjust perceived timing.
- `audio_cue` can route to future audio playback.

These hooks are future-facing only. First-pass acceptance does not require actual cinematic playback.

## Testing Strategy

Unit tests:

- `CombatPresentationEvent` stores typed fields and duplicates payload/tag data safely where needed.
- `CombatPresentationQueue` drains FIFO.
- `clear()` removes pending events.
- Disabled config filters expected event types.
- Snapshot diff produces damage, block, status, flash, and pulse events from known before/after states.
- Snapshot diff ignores unchanged or non-positive values.

Smoke tests:

- Combat screen still instantiates.
- Existing click fallback card play still works.
- Dragging an enemy-target card to an enemy plays the card.
- Invalid drag release does not mutate combat state.
- Dragging a self/no-target card upward plays it when valid.
- DebugOverlay can disable dragging without breaking click play.

Manual verification:

- Start a run and enter combat.
- Hover cards and see visible card feedback.
- Drag an attack card to an enemy and see target highlight, damage number, and flash.
- End turn and see enemy action feedback.
- Toggle presentation options in DebugOverlay and verify disabled effects stop showing.

## Review Requirements

After the implementation feature is complete, run the project-required two-stage review.

Stage 1: Spec Compliance Review

- Verify all files, scenes, scripts, nodes, signals, resources, input behavior, and debug controls match this spec and the implementation plan.
- Do not proceed to code quality review if drag fallback, queue behavior, snapshot diff, enemy turn feedback, or DebugOverlay toggles are missing.

Stage 2: Code Quality Review

- Check GDScript typing, signal usage, node paths, queue boundaries, snapshot diff maintainability, testability, and duplicated UI logic.
- Classify findings as Critical, Important, or Minor.
- Fix Critical and Important issues before acceptance.

## Acceptance Criteria

- Combat has a presentation event queue and presentation layer separate from core combat rules.
- `CombatEngine`, `EffectExecutor`, and status gameplay rules do not depend on presentation classes.
- Mouse drag can play valid enemy-target, player-target, and self/no-target cards.
- Existing click play flow still works.
- Invalid drag release does not mutate combat state.
- Player and enemy turns both produce visible feedback from state deltas.
- Floating numbers show damage, block, and status changes.
- Combatants flash or pulse when hit or when status layers change.
- Target highlighting appears during drag targeting.
- DebugOverlay can disable presentation categories for development.
- Presentation playback does not block gameplay input or require animations to finish.
- Existing local tests and Godot import check pass after implementation.
